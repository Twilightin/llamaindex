## Issues

### 🔴 Critical / 🟠 Major

#### 1. No dependency manifest exists

**Location**: project root — no `requirements.txt`, `pyproject.toml`, or lock file

Once packages are installed, there is no file tracking what was installed or at what version. This causes:

1. A second developer (or CI) cannot reproduce the environment.
2. Dependency pinning — the primary defence against supply-chain attacks — is impossible.
3. Security advisories against dependencies (e.g., `httpx`, `tiktoken`) cannot be audited.

**Fix**: Create `backend/requirements.txt` immediately, even if empty, and update `CLAUDE.md` to require regenerating it after every `pip install`:

```
# backend/requirements.txt
# Core
llama-index>=0.10.0,<0.11
openai>=1.0.0,<2.0

# Dev / testing
pytest>=8.0
python-dotenv>=1.0
```

Capture a lockfile after installing:
```bash
pip freeze > backend/requirements-lock.txt
```

#### 3. Two divergent code-reviewer agent definitions — no single source of truth

**Location**: `sample.md` vs `.claude/agents/code-reviewer.md`

Both files define the code-reviewer agent with different output formats, severity scales, and memory update instructions. When the agent is invoked, non-deterministic review quality results. The `sanyuan0704/code-review-expert` skill in `skills-lock.json` adds a third definition.
