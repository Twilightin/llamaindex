# Sub-Agent Cooperation Guide

This guide explains how to work with the sub-agent system in this project for maximum efficiency.

---

## The Core Idea

The biggest mistake people make with sub-agents is treating them as "workers who write code". This causes:

- **Isolated context** — each sub-agent starts fresh with no memory of what other agents did
- **Silent failures** — the main agent only sees "task done", not *what* was actually done
- **Unfixable bugs** — when something breaks, re-delegating opens a brand new session with zero history

**The correct mental model:**

| Role | Responsibility |
|------|---------------|
| **Main Claude** (you talk to this) | The sole executor. Writes all code. Has the full picture. |
| **Sub-agent** (`llamaindex-researcher`) | Expert consultant. Researches and plans only. Never touches source files. |
| **File system** (`.claude/task/*.md`) | The shared brain. Persistent memory across all agents and sessions. |

---

## The Workflow in Practice

### Before Writing Any Code — Use `/research`

```
You: /research add persistent ChromaDB vector store
```

What happens behind the scenes:
1. The `/research` skill reads `.claude/task/context.md` (current project state)
2. It launches `llamaindex-researcher` as a sub-agent
3. The researcher searches the latest LlamaIndex docs, checks the codebase, writes a detailed plan to `.claude/task/chroma_plan.md`, and updates `context.md`
4. The researcher returns a short 3-sentence summary to the main agent
5. You see the summary + plan file path

### Review the Plan

```
You: read the plan and show me the key steps
```

The main agent reads `.claude/task/chroma_plan.md` and summarises. You review. If it looks good:

```
You: looks good, go ahead
```

### Main Agent Implements Everything

The main agent reads the plan and writes all the code itself — never delegating the actual implementation back to a sub-agent. This is critical: **only one agent touches source files**, so it always has the complete picture of what was built.

## Concrete Examples

### Example 1 — Adding a New Feature

```
❌ Wrong:
You: "add ChromaDB persistent storage"
→ Claude tries to figure it out on the fly, may use wrong API, misses edge cases

✅ Correct:
You: "/research add ChromaDB persistent storage"
→ researcher finds the exact packages, import paths, and gotchas
→ you review the plan
You: "go"
→ main Claude implements cleanly from the plan
```

### Example 2 — You're Unsure Which Approach to Use

```
❌ Wrong:
You: "what's the best way to load PDFs?"
→ Claude answers from training data, which may be outdated (LlamaIndex API changed a lot)

✅ Correct:
You: "/research load and index PDF files"
→ researcher fetches the CURRENT docs, finds the right loader for LlamaIndex 0.10+
→ plan includes exact pip install commands and working code snippets
```

### Example 3 — Debugging

```
❌ Wrong:
You: "the vector store isn't persisting between runs, fix it"
→ Claude guesses at the fix without understanding the full context

✅ Correct:
You: "read context.md and the chroma_plan.md, then fix the persistence bug"
→ Main Claude reads both files, understands the original intent, fixes correctly
```

## The context.md File — Your Project Memory

`.claude/task/context.md` is the most important file in this workflow. It tracks:

- **What has been built** (current status checkboxes)
- **Key architecture decisions** (so future agents don't contradict them)
- **Active plans** (links to all plan files)
- **Progress log** (who did what and when)

**Keep it updated.** After a significant implementation, tell Claude:

```
You: update context.md to reflect that ChromaDB is now set up
```

This ensures the next sub-agent you launch has accurate context and doesn't re-research something already decided.

## When to Use `/research` vs. Just Asking Claude

| Situation | Use `/research`? |
|-----------|-------------|
| New feature you haven't built before | ✅ Yes |
| Unsure which LlamaIndex class/API to use | ✅ Yes |
| Feature that touches multiple files | ✅ Yes |
| Small bug fix in existing code | ❌ No — just ask Claude directly |
| Renaming a variable | ❌ No |
| Updating a config value | ❌ No |

Rule of thumb: **if you'd Google it first, use `/research` instead.**

## Why This Works

Sub-agents return only a **concise summary** (a few sentences) to the main conversation history, not hundreds of lines of research. This means:

- The main agent's context window stays clean and focused
- All the detail lives in `.md` files that can be re-read at any time
- When a bug occurs, the full implementation history is always accessible in the plan files
- New sessions start informed — just point Claude at `context.md`
